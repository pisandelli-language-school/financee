## 2023-10-27 - RBAC Authorization Bypass on Dynamic Backoffice API Routes
**Vulnerability:** The dynamic backoffice API routes (`server/api/backoffice/[section].ts` and `server/api/backoffice/[section]/[id].ts`) were protected using only `requireSupabaseUser`. This only validated authentication (the user is logged in) but failed to enforce authorization (whether the user has the required permissions to access the specific section or perform the requested action), leading to a significant authorization bypass.
**Learning:** Over-reliance on simple authentication checks (`requireSupabaseUser`) for dynamic, module-based routes can expose sensitive CRUD operations to unauthorized authenticated users. When an API route dynamically handles multiple modules and actions, the authorization logic must explicitly map those parameters to the underlying RBAC system.
**Prevention:** Replaced `requireSupabaseUser` with `requirePermission`. Implemented a `sectionModuleMap` and `methodActionMap` to dynamically construct the required permission key (e.g., `categorias.view`, `contas.create`) based on the requested route parameters and HTTP method, ensuring strict RBAC enforcement.
