## 2024-07-20 - Enforce RBAC on Dynamic Backoffice APIs
**Vulnerability:** The dynamic API routes `server/api/backoffice/[section].ts` and `[section]/[id].ts` used `requireSupabaseUser(event)` instead of `requirePermission(event, permissionKey)`. This created an authorization bypass where any authenticated user could view, create, update, or delete backoffice resources regardless of their actual permissions (Authentication vs Authorization Bypass).
**Learning:** When using dynamic routes that serve multiple domain entities, a static authorization check is not possible. Developers may fall back to simple authentication checks if they forget to implement a dynamic permission mapping for the route parameters.
**Prevention:** Always ensure `requirePermission` or `requireFinanceeUser` is used for API authorization rather than `requireSupabaseUser`. For dynamic routes, map the route parameters to their respective RBAC modules and enforce permissions based on the HTTP method.
