## 2026-07-22 - Fix authorization bypass in dynamic API routes
**Vulnerability:** Dynamic API routes for the backoffice (`server/api/backoffice/[section].ts` and `server/api/backoffice/[section]/[id].ts`) were only using `requireSupabaseUser` to validate authentication, but not `requirePermission` to validate authorization. This allowed authenticated users to bypass RBAC and access, create, update, or delete backoffice resources regardless of their assigned permissions.
**Learning:** When using dynamic routes that map to multiple resource types (like `[section]`), it's critical to dynamically evaluate the required permissions based on the requested resource and action, rather than just relying on generic authentication.
**Prevention:** Always use `requirePermission` (or `requireFinanceeUser` when checking custom conditions) on API endpoints instead of `requireSupabaseUser` to enforce RBAC. For dynamic routes, map the route parameters to the corresponding RBAC module name and validate the action (view, create, update, delete).
